# Hermes Workflow Contract

This file governs all Hermes agent sessions in this project. Every agent must read and comply with these rules.

## Mandatory rules

1. **Never assume project context** — always load it from files before acting.
2. **Bootstrap before everything** — no planning, coding, verification, or deployment before bootstrap completes.
3. **No implementation before proposal and design exist** — `proposal.md` and `design.md` must be present.
4. **No verification against chat intent** — verify against acceptance criteria in `proposal.md` only.
5. **No deployment without QA signoff** — `verification.md` must have `signoff: pass`.
6. **No self-authorized scope changes** — developers must escalate scope discoveries to the orchestrator.
7. **Delegated sub-agents must self-bootstrap** — do not rely on parent agent summaries for state.
8. **Record mistakes immediately** — do not batch mistake-log entries after the fact.

## State sources (in order of authority)

1. `~/coding-projects/project-map.yaml` — project location registry
2. `.ai/shared-memory/` — cross-session continuity layer
3. `openspec/changes/<change-id>/status.yaml` — per-change phase authority
4. `openspec/changes/<change-id>/handoff.md` — role-to-role baton
5. Current git branch + worktree

## Role hierarchy

- **Orchestrator**: advances phase state, routes work, enforces prerequisites
- **Product owner**: defines problem and acceptance criteria
- **Tech lead**: produces architecture design and tasks
- **Developer**: implements within assigned branch/worktree
- **QA engineer**: validates against acceptance criteria, distills lessons
- **Release manager**: ensures operational readiness and documents rollout

## Skill invocations

All workflow entry points use the `openspec-*` skill namespace:
- `openspec-orchestrator` — main entrypoint
- `openspec-bootstrap` — called by all roles before acting
- `openspec-<role>` — individual role skills
- `openspec-heartbeat` — autonomous coordination (runs via cron)
